Cloud Native Identity Platform

Browser → Nginx :80 → auth-service :8081 → PostgreSQL & Redis

How it works

1
Register a user
Creates a record in PostgreSQL DB Write
The /auth/register endpoint inserts username and password into the users table in PostgreSQL. If the username already exists, the DB returns a duplicate key error.
2
Login & get JWT
Verifies credentials, returns a signed token Token Issued
The /auth/login endpoint queries PostgreSQL for the user. If the password matches, JwtUtil creates a JWT signed with HMAC-SHA256. The token contains the username, issue time (iat), and expiration (exp). The token is returned to the browser.
3
Verify token
Checks signature & Redis blacklist Token Validated
The /auth/verify endpoint does three things: (1) Parses the Authorization: Bearer <token> header, (2) Checks if the token jti is in Redis blacklist:{jti} (would mean revoked), (3) Verifies the JWT signature and expiration. Returns the username if valid.
4
Logout (revoke token)
Adds token to Redis blacklist Redis Write
The /auth/logout endpoint stores the token in Redis with key blacklist:<token>. The next /auth/verify call will find this key and reject the token. This is how stateless JWT tokens are revoked without a server-side session store.
5
Verify again (should fail)
Token is now blacklisted Rejected
After logout, /auth/verify finds the token in the Redis blacklist and returns "token revoked". This demonstrates the complete lifecycle: register → login → access → logout → denied.

Register Step 1

Creates a user record in PostgreSQL. The password is stored directly (plain text in Phase 3; will be BCrypt-hashed in Phase 4).

Login Step 2

Authenticates against PostgreSQL. On success, auth-service signs a JWT with HMAC-SHA256 and returns it.

JWT Token Steps 3-5

The JWT is stored in your browser's localStorage. Use Verify Token to check its validity against auth-service. Login to get a token, Logout (above) to blacklist it in Redis.
