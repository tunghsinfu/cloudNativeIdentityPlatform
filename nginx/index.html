Cloud Native Identity Platform

Browser → Nginx :80 → auth-service :8081 → PostgreSQL & Redis

How it works

1
Register a user
Creates a record in PostgreSQL DB Write
The /auth/v1/register endpoint BCrypt-hashes the password and inserts username, hash, and role into the users table in PostgreSQL. If the username already exists, it returns a BAD_REQUEST error.
2
Login & get JWT
Verifies credentials, returns a signed token Token Issued
The /auth/v1/login endpoint queries PostgreSQL, verifies the password via BCryptPasswordEncoder.matches(), and issues a signed JWT (access token, 15 min) and an opaque refresh token (stored in Redis, 7 days). The JWT now includes jti (unique ID), sub (username), and role claims.
3
Verify token
Checks signature & Redis blacklist Token Validated
The /auth/v1/verify endpoint does three things: (1) The JwtAuthenticationFilter parses the Authorization: Bearer <token> header, (2) Checks if the token's jti is in Redis blacklist:{jti} (would mean revoked), (3) Sets the Spring Security context with username and role. The controller returns these from the authenticated context.
4
Logout (revoke token)
Adds token to Redis blacklist Redis Write
The /auth/v1/logout endpoint extracts the jti from the SecurityContext (set by JwtAuthenticationFilter) and stores it in Redis as blacklist:{jti}. The next /auth/v1/verify call will find this key and reject the token. This is how stateless JWT revocation works.
5
Verify again (should fail)
Token is now blacklisted Rejected
After logout, /auth/verify finds the token in the Redis blacklist and returns "token revoked". This demonstrates the complete lifecycle: register → login → access → logout → denied.
6
Explore: Rate Limit & Cache
Trigger 5 failed logins → 429 lockout • See Redis cache on success
Rate Limit — enter wrong password 5 times in Login. Watch the fail counter (N/5) next to the button. On the 6th attempt, the button disables for 60s and you get a red "Login locked" message.
Cache-Aside — after login succeeds, the interpretation panel shows (from Redis cache) or (from PostgreSQL). The cache is stored as JSON in Redis key user:{username} with TTL 300s.
Redis commands to inspect manually: docker compose exec redis redis-cli GET "login_fail:alice"
docker compose exec redis redis-cli GET "user:alice"
docker compose exec redis redis-cli TTL "user:alice"

Register Step 1

Creates a user record in PostgreSQL. The password is BCrypt-hashed by BCryptPasswordEncoder before storage.

Login Step 2

Authenticates against PostgreSQL (or Redis cache) using BCrypt. On success, returns an access token (JWT, 15 min) + refresh token (opaque UUID in Redis, 7 days). Rate limit: 5 wrong attempts → 60s lockout (N/5).

JWT Token Steps 3-5

The JWT is stored in your browser's localStorage. Use Verify to check validity, Refresh to get a new access/refresh token pair. Login to get a token, Logout (above) to blacklist it in Redis.
